# 参与 FCS 改进

FCS 的目标是被广泛使用、共同完善。任何人都可以提议改进——无论你是技术人员、自由职业者、HR、律师，还是只是用过 FCS 想分享感受的人。

## 提改进建议的方式

### Issue（适合讨论）
开 Issue 来：
- 反馈你在使用中遇到的具体场景（协议没覆盖、产生歧义、引发了纠纷）
- 提出新章节的想法
- 询问某条款的本意
- 分享实践案例

### Pull Request（适合具体修改）
提 PR 来：
- 修正错别字、表述歧义
- 增补条款（建议先开 Issue 讨论，避免重复劳动）
- 提交翻译版本

## 改进的原则

提改进建议时，请尽量遵守以下原则：

1. **简洁优先**：能用一句话说清的不写两句
2. **方法 > 细则**：规定方向，避免锁死具体格式
3. **包容**：考虑不同行业、不同团队规模的适用性
4. **保留人文气**：FCS 不是冰冷的合约，措辞应温和、可读
5. **稳定性**：核心结构（贡献度量、退出与加入、争议处理）调整需慎重，影响所有已采纳团队

## 版本号规则

采用语义化版本：

| 变更类型 | 版本号变化 | 例子 |
|---|---|---|
| 重大结构变化 | 大版本 +1 | v1.0 → v2.0 |
| 增删条款、不破坏旧版本兼容 | 小版本 +1 | v1.0 → v1.1 |
| 勘误、措辞优化 | 补丁号 +1 | v1.0 → v1.0.1 |

## 翻译

欢迎贡献其他语言版本。翻译文件命名约定：
- 英文：`protocol.en.md`
- 日文：`protocol.ja.md`
- 其他语言：`protocol.{ISO639 code}.md`

放置在仓库根目录，并在 README 中添加链接。

## 行为准则

- 友善、专业、对事不对人
- 假设善意：对方的反对可能源于不同的实践经验，不一定是反对你本人
- 数据与案例比抽象论证更有说服力
- 即便改进未被采纳，也欢迎你 fork 一个变体版本——FCS 鼓励百花齐放
